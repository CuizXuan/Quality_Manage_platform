# 质量保障平台 - 总体设计方案

## 一、项目定位与愿景

### 1.1 定位

从**单点 API 调试工具**演进为**覆盖开发、测试、运维全流程的质量保障平台**。

### 1.2 愿景

> 让每一个接口都有据可查、有测可依、有迹可循。

### 1.3 核心价值

| 角色       | 痛点                     | 解决方案               |
| :--------- | :----------------------- | :--------------------- |
| **开发**   | 接口文档滞后、联调困难   | 调试即文档，实时同步   |
| **测试**   | 手工测试低效、用例难复用 | 调试转用例，一键自动化 |
| **运维**   | 线上问题定位慢           | 历史快照对比，快速复现 |
| **管理者** | 质量状态不透明           | 数据大盘，趋势可见     |

### 3.2 模块详细说明

#### 3.2.1 请求调试（核心能力，已实现）

| 功能点          | 现状           | 演进方向                              |
| :-------------- | :------------- | :------------------------------------ |
| cURL/Fetch 解析 | ✅ 已实现       | 支持更多格式（Postman、Swagger 导入） |
| 请求编辑        | ✅ 已实现       | 增加环境变量提示、语法高亮            |
| 响应展示        | ✅ 已实现       | 增加 JSON Schema 校验、历史对比       |
| 历史记录        | ✅ LocalStorage | 迁移至数据库，支持云端同步            |

#### 3.2.2 用例管理（Phase 1 重点）

| 功能点          | 设计说明                                         |
| :-------------- | :----------------------------------------------- |
| **用例 CRUD**   | 将调试好的请求保存为可复用用例                   |
| **集合/文件夹** | 树形结构组织用例，支持拖拽排序                   |
| **环境变量**    | 多环境切换，变量语法 `{{variable}}`              |
| **断言**        | 支持状态码断言、响应体 JSONPath 断言、响应头断言 |
| **脚本**        | 前置脚本（生成签名）、后置脚本（提取变量）       |

#### 3.2.3 自动化测试（Phase 2-3 重点）

| 功能点       | 设计说明                                                |
| :----------- | :------------------------------------------------------ |
| **场景编排** | 多用例串联执行，支持变量传递（如登录 token → 后续请求） |
| **数据驱动** | 绑定 CSV/JSON 数据源，遍历执行                          |
| **定时任务** | Cron 表达式配置，周期性执行并告警                       |
| **压力测试** | 并发请求、阶梯加压、实时 TPS/RT 监控                    |

#### 3.2.4 单元测试与覆盖率集成（Phase 3 重点）

| 功能点             | 设计说明                                                   |
| :----------------- | :--------------------------------------------------------- |
| **覆盖率采集**     | 对接 JaCoCo（Java）、Istanbul（JS）、Coverage.py（Python） |
| **覆盖率报告解析** | 解析标准 `lcov` / `cobertura` 格式，入库存储               |
| **用例-代码关联**  | 通过注解/路径规则，建立 API 用例与代码的映射               |
| **覆盖率看板**     | 展示每个接口对应代码的覆盖率，高亮未覆盖行                 |

#### 3.2.5 缺陷管理（Phase 3 重点）

| 功能点           | 设计说明                                          |
| :--------------- | :------------------------------------------------ |
| **一键提缺陷**   | 从测试结果页一键创建缺陷，自动填充请求/响应快照   |
| **轻量缺陷看板** | 内置简单的缺陷工作流（待处理→处理中→已解决→关闭） |
| **第三方同步**   | 插件化对接 Jira/TAPD，双向同步状态                |

#### 3.2.6 文档与报告（Phase 2-4）

| 功能点           | 设计说明                                 |
| :--------------- | :--------------------------------------- |
| **接口文档生成** | 基于用例自动生成 Markdown/HTML 文档      |
| **测试报告**     | 单次执行报告、定时任务报告、版本质量报告 |
| **质量趋势**     | 通过率趋势、响应耗时趋势、缺陷收敛趋势   |

#### 3.2.7 团队协作（Phase 4）

| 功能点       | 设计说明                         |
| :----------- | :------------------------------- |
| **多用户**   | 用户注册/登录，RBAC 权限模型     |

### 4.2 核心表结构

sql

```
-- 用例表
CREATE TABLE test_case (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name VARCHAR(200) NOT NULL,
    method VARCHAR(10) NOT NULL,
    url TEXT NOT NULL,
    headers TEXT,
    params TEXT,
    body TEXT,
    body_type VARCHAR(20) DEFAULT 'json',
    collection_id INTEGER,
    assertions TEXT,          -- JSON 数组，断言规则
    pre_script TEXT,          -- 前置脚本
    post_script TEXT,         -- 后置脚本
    created_by INTEGER,
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);

-- 执行记录表
CREATE TABLE execution_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    case_id INTEGER,
    scenario_id INTEGER,
    status VARCHAR(20),
    request_snapshot TEXT,     -- 请求快照（JSON）
    response_status INTEGER,
    response_headers TEXT,
    response_body TEXT,
    response_time_ms INTEGER,
    assertion_results TEXT,    -- 断言结果
    error_message TEXT,
    created_at TIMESTAMP
);

-- 场景表
CREATE TABLE scenario (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name VARCHAR(200) NOT NULL,
    description TEXT,
    variables TEXT,            -- 场景级变量
    created_by INTEGER,
    created_at TIMESTAMP
);

-- 场景步骤表
CREATE TABLE scenario_step (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    scenario_id INTEGER,
    case_id INTEGER,
    step_order INTEGER,
    extract_rules TEXT,        -- 变量提取规则
    FOREIGN KEY (scenario_id) REFERENCES scenario(id),
    FOREIGN KEY (case_id) REFERENCES test_case(id)
);

-- 环境表
CREATE TABLE environment (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name VARCHAR(50) UNIQUE NOT NULL,
    variables TEXT,
    is_default BOOLEAN DEFAULT 0,
    created_by INTEGER
);

-- 缺陷表（轻量级）
CREATE TABLE defect (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title VARCHAR(500) NOT NULL,
    description TEXT,
    severity VARCHAR(20),      -- critical/high/medium/low
    status VARCHAR(20),        -- open/in_progress/resolved/closed
    execution_log_id INTEGER,  -- 关联的执行记录
    assigned_to INTEGER,
    created_by INTEGER,
    created_at TIMESTAMP,
    resolved_at TIMESTAMP
);

-- 代码覆盖率表
CREATE TABLE code_coverage (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    case_id INTEGER,
    file_path VARCHAR(500),
    line_coverage FLOAT,
    branch_coverage FLOAT,
    uncovered_lines TEXT,      -- JSON 数组
    report_date DATE,
    FOREIGN KEY (case_id) REFERENCES test_case(id)
);
```

## 五、分阶段实施路线图

### Phase 1：基础平台化（4 周）

**目标**：从单机工具升级为有数据持久化的基础平台

| 周次   | 核心任务              | 关键产出                             |
| :----- | :-------------------- | :----------------------------------- |
| Week 1 | 数据库迁移 + 用例管理 | SQLite 存储、用例 CRUD、数据迁移工具 |
| Week 2 | 环境管理 + 断言机制   | 多环境切换、变量替换、基础断言       |
| Week 3 | 场景编排雏形          | 多用例串联、变量提取与传递           |
| Week 4 | CLI 工具 + CI 集成    | 命令行执行、JUnit 报告输出           |

**里程碑**：可通过 CLI 在 CI 流水线中运行用例集。

------

### Phase 2：自动化测试增强（4 周）

**目标**：支持数据驱动和定时任务，形成自动化回归能力

| 周次     | 核心任务     | 关键产出                              |
| :------- | :----------- | :------------------------------------ |
| Week 5-6 | 数据驱动测试 | CSV/JSON 数据源绑定、参数化执行       |
| Week 7   | 定时任务系统 | Cron 配置、任务调度、执行历史         |
| Week 8   | 测试报告增强 | 趋势图表、失败原因分析、邮件/钉钉通知 |

**里程碑**：核心业务场景实现 7x24 小时自动化巡检。

------

### Phase 3：质量体系融合（4 周）

**目标**：接入单元测试覆盖率，打通缺陷管理

| 周次      | 核心任务     | 关键产出                                  |
| :-------- | :----------- | :---------------------------------------- |
| Week 9-10 | 覆盖率集成   | 解析覆盖率报告、用例-代码关联、覆盖率看板 |
| Week 11   | 缺陷管理模块 | 一键提缺陷、轻量看板、Jira 插件           |
| Week 12   | 质量门禁     | CI 中集成覆盖率阈值检查、用例通过率门禁   |

**里程碑**：形成「代码提交 → 单测 → 覆盖率 → API 测试 → 缺陷闭环」的完整链路。

------

### Phase 4：团队协作与智能化（6 周）

**目标**：支持多用户协作，引入智能分析能力

| 周次       | 核心任务     | 关键产出                                 |
| :--------- | :----------- | :--------------------------------------- |
| Week 13-14 | 多用户与权限 | 用户系统、RBAC、团队空间                 |
| Week 15-16 | 质量大盘     | 多维度数据可视化、项目质量评分           |
| Week 17-18 | 智能分析     | 失败模式聚类、变更影响推荐、智能告警降噪 |

**里程碑**：平台具备团队级服务能力，覆盖开发、测试、运维全角色。

## 七、非功能性设计

### 7.1 性能指标

| 指标                 | 目标值                  |
| :------------------- | :---------------------- |
| 单用例执行响应       | < 100ms（不含网络请求） |
| 场景执行（10 步骤）  | < 5s                    |
| 并发执行（100 用例） | < 30s                   |
| 前端首屏加载         | < 2s                    |

### 7.2 安全设计

| 层面     | 措施                                   |
| :------- | :------------------------------------- |
| 敏感数据 | 请求头/响应中的敏感字段支持脱敏配置    |
| 数据隔离 | 多用户数据隔离，团队空间权限控制       |
| 审计日志 | 关键操作（删除用例、修改环境）记录日志 |
| API 安全 | JWT 认证、请求频率限制                 |

### 7.3 扩展性设计

- **插件化缺陷系统**：定义标准接口，适配多种第三方系统
- **报告模板**：支持自定义报告模板（Jinja2）
- **脚本引擎**：支持 Python/JavaScript 脚本扩展
